<objective>
Fix three bugs in the transaction/account balance system:
1. D-01: Non-recurring transactions not deducted from balance on creation
2. D-02: Transaction value updates not adjusting balance  
3. D-03: Recurring transactions need confirmation mechanism before debiting

The solution requires atomic balance synchronization in the service layer, plus a manual confirmation workflow for recurring payments.
</objective>

Add new method ConfirmRecurring to TransactionsService:
1. Fetch transaction: t, err := s.txnsRepo.GetByID(transactionID)
2. Verify t.IsRecurring == true, return error if not
3. Fetch account: acc, err := s.accRepo.GetByID(t.AccountID)
4. Begin atomic transaction
5. Calculate new balance: newBalance := acc.CurrentBalance + t.Amount (t.Amount is negative for debit)
6. Update account balance: s.accRepo.UpdateBalance(t.AccountID, newBalance, tx)
7. Advance next_occurrence (same logic as RecordDebit):
   - weekly: next = current + 7 days
   - bi-weekly: next = current + 14 days  
   - monthly: next = AddMonthClamped(current, 1)
   - yearly: next = AddMonthClamped(current, 12)
8. Call s.txnsRepo.AdvanceNextOccurrence(id, next, tx)
9. Commit transaction
10. Return new next_occurrence in response so UI can update
  </action>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-01 | Spoofing | POST /transactions/:id/confirm | mitigate | Validate transaction exists and belongs to user's account before confirming |
| T-09-02 | Tampering | Balance calculation | mitigate | Use atomic SQL transaction (not two-phase commit) — if balance update fails, transaction insert rolls back |
| T-09-03 | Repudiation | ConfirmRecurring audit | mitigate | Log before/after balance and next_occurrence for audit trail |
| T-09-04 | Information Disclosure | Confirm endpoint | accept | Transaction ID is UUID, low-risk disclosure |
| T-09-05 | DoS | Multiple rapid confirms | mitigate | Rate limit per transaction ID, check next_occurrence > now before advancing |
| T-09-06 | Elevation | Transaction amount | mitigate | Validate amount is reasonable (non-NULL, within bounds) in handler before passing to service |
</threat_model>
