## Spacing Scale

Declared values (multiples of 4 only). All spacing follows the 8-point grid already established by the existing pages.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps (`gap-1`), badge padding |
| sm | 8px | Tight element spacing (`gap-2`, `px-2`) |
| md | 16px | Default element spacing (`gap-4`, `px-4`) |
| lg | 24px | Section padding (`py-6`) |
| xl | 32px | Layout gaps (`gap-8`) |
| 2xl | 48px | Major section breaks (`py-12`) |
| 3xl | 64px | Page-level spacing (`py-16`) |

Exceptions:
- AccountSelector height: `h-7` (28px) — matches existing component, do not change
- Form input height: `h-8` (32px) for selects, `h-11` (44px) for primary inputs — matches transactions page pattern

## Surface Specifications

### Surface 1 — Settings Page (`web/src/pages/settings.tsx`)

Layout: `max-w-2xl mx-auto px-4 sm:px-6 py-8 flex flex-col gap-4` — identical to TransactionsPage and AccountsPage container. Do not deviate.

#### Section: Account context

- Reuse `AccountSelector` component as-is (`web/src/components/AccountSelector.tsx`)
- Place in a header row alongside the page title:
  ```
  [h1: Pay Schedules]   [AccountSelector dropdown]
  ```
- Same flex row pattern as TransactionsPage: `flex flex-col sm:flex-row items-start sm:items-center justify-between gap-4`
- Sub-label below h1: `text-xs text-muted-foreground` — "Account: {name}" (matches TransactionsPage pattern)

#### Section: Schedule list

- Each schedule rendered as a `<Card>` with `<CardContent className="py-4">`
- Row layout: 3-column grid on sm+, stacked on mobile — `grid grid-cols-1 sm:grid-cols-3 gap-4 items-center`
- Column 1 (label + frequency):
  - Primary text: label if set, otherwise frequency — `font-medium`
  - Secondary: frequency + anchor date — `text-xs text-muted-foreground`
  - Semi-monthly: show both day_of_month and day_of_month_2 in secondary text
- Column 2 (amount):
  - `font-medium tabular-nums text-right` — formatted as `+$X,XXX.00` in green (`text-green-600` — matches transaction amount display)
- Column 3 (actions):
  - Edit button: `<Button variant="ghost" size="sm">` with `<Edit2 size={14} />`
  - Delete button: `<Button variant="ghost" size="sm">` with `<Trash2 size={14} />`
  - Right-aligned: `flex gap-2 justify-end`

#### Section: Add schedule CTA

- Positioned in page header row (right side), same position as "New Transaction" in TransactionsPage
- Label: "Add Schedule"
- Component: `<Button size="sm"><Plus size={16} /> Add Schedule</Button>`

#### Section: Inline add/edit form

- Appears as a `<Card>` appended below the schedule list (same as TransactionsPage inline form)
- `<CardHeader>` with `<CardTitle className="text-base">`: "New Schedule" or "Edit Schedule"
- Form fields (in order):

  | Field | Input type | Required | Notes |
  |-------|-----------|----------|-------|
  | Label | `<Input>` text | No | Placeholder: "e.g. Main paycheck" |
  | Frequency | `<select>` | Yes | Options: Weekly, Bi-weekly, Semi-monthly, Monthly |
  | Anchor Date | `<Input type="date">` | Yes | ISO date picker |
  | Amount | `<Input type="number" step="0.01">` | Yes | Dollar display, cents storage — same pattern as TransactionsPage |
  | Day of month | `<Input type="number" min="1" max="31">` | Conditional | Shown only when frequency = monthly or semi-monthly |
  | Day of month 2 | `<Input type="number" min="1" max="31">` | Conditional | Shown only when frequency = semi-monthly |

- Form field label style: `<label className="block text-xs font-medium mb-2">`
- Frequency select style: `className="w-full h-8 px-2.5 py-1 rounded-lg border border-input bg-transparent text-base md:text-sm"` — matches TransactionsPage
- Form action buttons: `<div className="flex gap-2 pt-4">` with primary submit + outline Cancel

#### Loading skeleton

- 3 skeleton rows while fetching: `<div className="h-16 rounded-xl bg-card/60 animate-pulse border border-border/40" />` — matches TransactionsPage pattern

### Surface 2 — WAIT Verdict (`web/src/components/CheckWidget.tsx`)

#### State machine extension

Current states: `'idle' | 'loading' | 'verdict'`
No new states needed. The verdict state already handles both YES and NO; WAIT is a third branch within verdict rendering.

Detection logic: `result.will_afford_after_payday === true && result.can_buy === false`

#### WAIT verdict card

The verdict `<motion.div>` already uses `initial={{ scale: 0.90, opacity: 0 }}` — apply identical animation to WAIT. No changes to animation spec.

Inline style for WAIT card:
```
background: 'var(--color-verdict-wait-tint)'
borderColor: 'oklch(0.78 0.17 85 / 0.35)'
boxShadow: '0 0 32px oklch(0.78 0.17 85 / 0.25)'
```

Display text (36px bold):
```
color: 'var(--color-verdict-wait)'
```

Verdict word displayed: `"WAIT"` — all caps, same `text-4xl font-bold tracking-tight leading-none` as YES/NO

Sub-text line (14px, `text-foreground/80`):
```
Not yet — you'll have enough after {waitUntil}
```
Where `{waitUntil}` is the `wait_until` API field formatted as `"Apr 10"` using `new Date(result.wait_until).toLocaleDateString('en-US', { month: 'short', day: 'numeric' })`.

Risk badge for WAIT: use the new risk level string `"WAIT"`. Add to `RISK_COLORS` map:
```ts
WAIT: 'var(--color-verdict-wait)',
```
Badge label: `"WAIT RISK"` — same `text-[10px] font-semibold uppercase tracking-widest px-2 py-1 rounded-md w-fit` pattern.

Purchasing power and buffer remaining lines: show normally (same as YES/NO).

## Interaction Contracts

### Settings — Schedule CRUD

| Action | Trigger | Optimistic? | On success | On error |
|--------|---------|-------------|------------|---------|
| Add schedule | "Add Schedule" button click | No | `toast.success('Schedule added')` + invalidate `['pay-schedules', accountId]` | `toast.error('Failed to add schedule')` |
| Edit schedule | Edit icon button click | No | Form pre-filled; submit → `toast.success('Schedule updated')` + invalidate | `toast.error('Failed to update schedule')` |
| Delete schedule | Trash icon button → `window.confirm(...)` | No | `toast.success('Schedule deleted')` + invalidate | `toast.error('Failed to delete schedule')` |
| Cancel form | "Cancel" button or Escape | — | Form closes, list unchanged | — |

Query key pattern: `['pay-schedules', accountId]` — invalidated after every mutation.

Account context: read from `AccountContext` (same as TransactionsPage) — `const { selectedAccountId } = useContext(AccountContext)`

### WAIT Verdict — Interaction

- Clicking "Check another" resets to idle (no change from current reset behavior)
- WAIT card is not clickable / has no drill-down action
- The WaitUntil date is display-only text, not a link
