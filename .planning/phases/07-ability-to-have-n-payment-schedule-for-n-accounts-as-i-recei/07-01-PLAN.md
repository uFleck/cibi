        return "MEDIUM"

## Trust Boundaries

|----------|-------------|
| Service → Repo | Service layer trusts repo to return correctly typed data; repo trusts caller-provided UUIDs are valid |
| Engine → DB | Engine trusts SumUpcomingObligations to correctly filter by time window; must not compute obligations in application memory |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Tampering | `PayScheduleService.CreatePaySchedule` | mitigate | `accRepo.GetByID(accountID)` called before insert — prevents orphaned schedules with non-existent account_id; FK constraint at DB level as secondary guard |
| T-07-02 | Tampering | `amount` field on PaySchedule | mitigate | `validate:"min=0"` on handler request struct (enforced in Plan 07-02); service layer receives int64 — no float parsing in service, only in handler |
| T-07-03 | Elevation of privilege | No auth in this application | accept | Application is single-user, accessed via Tailscale; no authentication layer planned; accepted per project architecture |
| T-07-04 | Denial of service | Multiple schedules — unbounded list | accept | Personal finance tool; account owner controls schedules; no public endpoint; N schedules per account is practically bounded |
| T-07-05 | Information disclosure | Engine returning wrong account data | mitigate | `CanIBuyIt(accountID uuid.UUID, ...)` — every DB query is scoped to `accountID`; no cross-account leakage possible in single-user app |
